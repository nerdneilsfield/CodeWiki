### 当前问题诊断（针对你列出的失败模式）

**(A) Evidence（来自你提供的现有流程描述）**

* 模块聚类主要由 LLM 完成，输入偏向“核心组件 + 简化图信息”，存在语义漂移风险（阶段 3 描述）。

* 文档生成把大量源码直接塞进 prompt，递归分裂依赖 LLM 决策（阶段 4 描述），易造成重复、跑题、幻觉与不一致。
* Mermaid/Math/Link 修复以正则或 LLM 兜底（阶段 5 描述），缺少强校验与回归门禁。

**(B) Inference（问题根因推断）**

* **模块树乱/命名差**：LLM 在缺少稳定先验（目录/包/导出 API）与缺少可验证约束（schema+一致性检查）时容易输出不稳定结构。
* **漏组件/漏关系**：符号抽取与调用解析偏“文件内、浅层、可解析才记录”，导致图稀疏且偏差大，聚类与生成都在“坏图”上工作。
* **重复/冗长**：每个叶模块重复解释公共概念；缺少全局术语表/去重策略/引用规则。
* **幻觉/不符代码**：没有强制“证据引用”，也没有自动验证（例如：声明某函数存在/参数列表/调用链）
* **Mermaid/Math 坏、链接不一致**：缺少语法级 lint 与构建期 fail-fast；链接缺少全局规范化与锚点一致性策略。

## 2. 现有 CodeWiki 流程回顾（简要）

> 这里按你给的阶段描述，用“一句话：输入/输出/风险点”总结。

1. **语言检测**：输入文件列表 → 输出语言分布；风险：扩展名启发式对混合仓/生成目录不鲁棒。
2. **依赖图构建**：输入源码 → 输出组件集合 V 与关系 E；风险：组件抽取浅、跨文件调用解析弱、未解析边被丢弃导致图偏稀疏。
3. **LLM 聚类**：输入核心组件 + 图摘要 → 输出模块树；风险：结构不稳定、命名随意、覆盖不足、难以复现。
4. **递归文档生成**：输入模块树节点 → 输出 Markdown；风险：token 溢出引发 LLM 分裂漂移、内容重复/跑题/幻觉、跨页链接不一致。
5. **后处理与站点构建**：修复 math/mermaid/link → 生成静态 HTML；风险：缺少强校验与回归门禁，坏图坏公式可能漏过。

## 3. 总体架构升级方案（Pipeline级）

### 3.1 离线索引层（Index Layer）

**(A) Evidence**

* 现流程已有 AST/tree-sitter 解析思路与缓存概念，但索引产物较少，且关系解析偏弱（你提供的阶段 2/4 描述）。

**(B) Inference — 推荐默认策略（质量优先）**
构建一个“可增量、可追溯、可检索”的离线索引层，输出以下一等公民产物（都要可版本化）：

* **文件清单**：路径、语言、大小、hash、是否生成/第三方、是否测试文件。
* **语言分布**：用于选择语言 indexer 与资源分配。
* **符号表（Symbol Table）**：

  * `symbol_id`（稳定 ID）、`qualified_name`、`kind`（class/function/method/...）、定义位置（文件+range）、签名、docstring/注释、可见性、是否导出/公共 API。
* **依赖摘要**：import/export、extends/implements、calls、references（含 unresolved）。
* **README/配置摘要**：按段落切分的 README、构建脚本、CI 配置、lint/format 配置（用于 Getting Started / Best Practices）。

**向量库（可选但推荐）**

* 嵌入对象建议分层：

  1. README 段落/贡献指南/架构文档段落
  2. 文件级摘要（非源码全文）
  3. 符号“组件卡片”（signature + docstring + 关键边摘要）
  4. 典型调用片段（callsite 周围少量行）
* 检索策略：先 **结构过滤**（模块/目录/语言/导出 API），再 **语义召回**（embedding），最后 **证据裁剪**（只取必要片段进 LLM）。

**(C) Alternatives（快/省与其他路线）**

1. **快/省**：仅做文件清单 + import 图 + 粗符号（函数/类名、行号、docstring），不做 call resolution。
2. **中等**：对强类型语言（TS/Go/Java）用官方分析器做符号与引用，对 Python 只做 import/定义与启发式 calls。
3. **重型高精度**：引入语义索引格式（SCIP/LSIF 类）统一跨语言“定义/引用/实现”能力（见 3.2）。

**(D) Limits（限制/风险）**

* 多语言全精度解析代价高；需要按语言分级（“高价值语言优先”）。
* 需要处理生成目录/第三方依赖目录，否则索引会被噪声淹没。
* 向量库引入后要做可解释性与可回放（存 query 与命中证据），否则难调试。

**外部来源（工具参考，访问日期 2026-03-04）**

* Tree-sitter：增量解析与多语言语法树基础能力（官方站点与 GitHub）。([Tree-sitter][1])
* SCIP：语言无关的代码索引协议，可表达定义/引用等语义关系（Sourcegraph）。([GitHub][2])
* LSIF：离线持久化语言服务知识以支持“无运行分析器”的导航（Microsoft）。([GitHub Microsoft][3])

### 3.2 图构建与消歧层（Graph Layer）

**(B) Inference — 多语言统一组件模型（Component schema）**
把图从“临时结构”升级为“可验证的数据产品”。建议最小统一 schema：

```json
{
  "symbol": {
    "symbol_id": "lang:path#qualified_name(kind)",
    "lang": "Python|TypeScript|Go|...",
    "kind": "package|module|file|class|interface|function|method|type|variable|constant|macro",
    "name": "shortName",
    "qualified_name": "a.b.C.m",
    "file": "src/a/b.py",
    "range": {"start_line": 10, "start_col": 0, "end_line": 42, "end_col": 1},
    "signature": "m(x: int) -> str",
    "visibility": "public|protected|private|internal|unknown",
    "export_status": "exported|not_exported|unknown",
    "doc": {"docstring": "...", "comments": ["..."]},
    "hash": "def-snippet-hash"
  },
  "edge": {
    "edge_id": "stable-hash",
    "type": "imports|exports|calls|extends|implements|overrides|references|reads|writes|tests",
    "from": "symbol_id",
    "to": {"resolved_symbol_id": "symbol_id", "unresolved_ref": "rawNameOrPath"},
    "evidence_refs": [{"file": "x", "range": {"start_line": 1, "start_col": 0, "end_line": 1, "end_col": 10}}],
    "confidence_level": "high|medium|low",
    "resolver": "ast|treesitter|tsc|pyright|go/packages|heuristic"
  }
}
```

**关键机制：消歧 + 置信度 + 保留未解析**

* 不再“解析不到就丢弃”。未解析引用也应进入图（`unresolved_ref` + low confidence）。
* 任何可解析边必须带 `evidence_refs`（文件+范围），为后续“证据驱动写作”提供可引用锚点。

### 3.3 聚类与模块树层（Clustering Layer）

**(B) Inference — 推荐策略：图预聚类 + LLM 语义命名（LLM 不做全量聚类）**

* **预聚类**（确定结构骨架）：

  1. 目录/包先验（package boundary）
  2. SCC（强连通分量）保证循环依赖同簇
  3. 社区发现（如 Leiden）在“加权模块图”上做凝聚（加权边：imports/calls/exports/test-dep）
* **LLM 的职责**：

  * 给簇命名、写简介、在“少量明确冲突点”上做边界调整（受严格约束）
  * 输出可解析 JSON，且必须满足 determinism 规则（固定排序、固定 ID 生成规则）

**外部来源（算法参考，访问日期 2026-03-04）**

* Leiden 社区发现相对 Louvain 的连通性保证与质量特性（Scientific Reports 论文）。([Nature][4])

### 3.4 生成与一致性层（Generation Layer）

**(B) Inference — 两阶段写作 + 全局资产 + 缓存**

* 阶段 A：先产出**模块概览草稿**（每模块 1 页，主要列 API、关键符号、关键依赖边、典型入口/数据流图）
* 阶段 B：再产出**组件细节**（叶子模块）
* 共享资产（全站一致性核心）：

  * **重复控制**（公共概念只在“概念页/术语表”解释一次，其他页面引用链接）
* 缓存：以 `(component_hash, index_version, prompt_version, model_version)` 为键，支持增量重建。

### 3.5 后处理与QA层（Post Layer）

**(B) Inference — 从“修修补补”升级为“lint + 修复 + 回归门禁”**

* Mermaid：语法/渲染 lint（编译成 SVG/PNG）→ 失败则自动修复 → 再失败则降级为代码块并标注错误原因。

  * 参考：mermaid-cli 可从定义生成 svg/png/pdf（GitHub）。([GitHub][5])
* Math：用 KaTeX 渲染并捕获错误（支持 `throwOnError=false` 的降级策略），失败同样走修复/降级。([KaTeX][6])
* Link：对本地链接与锚点做强校验（remark-validate-links），外链用 lychee 做断链检查。([GitHub][7])
* 回归：保留 lint 日志与差异报告（模块树 diff、链接变化、失败的图/公式列表）。

## 4. 五大决策面逐项优化（逐项：默认策略/备选/失败模式/验证）

### 4.1 Indexing & Chunking

**(A) Evidence**

* 现流程按文件逐个解析并直接把组件源码塞进 prompt；缺少“组件卡片/索引摘要”的中间层（你提供的阶段 2/4）。

**(B) Inference — 推荐默认策略（质量优先）**

1. **分层索引产物**（从“源码全文”转到“可控摘要”）：

   * `Symbol Table`：定义/签名/docstring/导出状态/证据范围
   * `Symbol Card`（给 LLM 用的轻量摘要）：

     * signature + docstring（如有）
     * 关键边摘要（top imports/calls/implementations，附 evidence_ref）
     * 关键代码摘录（仅定义头部与关键分支附近少量行，带行号范围）
2. **Chunking 原则**：

   * 文档生成的 chunk 单位优先是“符号定义 + 关联证据”，而不是固定 token 大小。
   * embeddings 的 chunk 单位优先是：README 段落、文件级摘要、符号卡片、调用片段。
3. **检索计划（Retrieval Plan）**：

   * Step 1：结构过滤（模块/目录/语言/导出 API/测试）
   * Step 2：语义召回（embedding：query = 模块名+关键符号+职责）
   * Step 3：证据裁剪（只保留能支撑文档断言的最小片段）

**(C) Alternatives（至少三角度）**

* 角度 1（最省）：不用向量库，仅用路径规则 + BM25（grep/ripgrep + 简单倒排）。
* 角度 2（折中）：向量库只嵌入 README/注释/符号卡片，不嵌入源码行，降低噪声。
* 角度 3（最强）：引入语义索引格式（SCIP/LSIF）作为符号/引用事实源，embedding 只做“语义召回”，事实以索引为准。([GitHub][2])

**失败模式与回退（Failure modes & fallbacks）**

* **召回不足**（漏关键符号）：回退到结构遍历（模块内导出符号全列）+ top-k 依赖边扩展。
* **召回过多**（噪声大导致 LLM 乱写）：增加结构过滤（仅 public API + 关键入口 + 与本模块耦合最高邻居）。
* **索引漂移**（增量更新出错）：提供“全量重建”开关与索引一致性校验（hash 对齐）。

**Metrics & evaluation plan（不造数，只给采集方法）**

* 索引覆盖：统计可解析文件比例、符号数量趋势、unresolved 边比例趋势。
* 检索质量：记录每次生成的命中文档证据集合（symbol_id 列表）与最终引用覆盖率。
* 成本：按阶段记录（索引/聚类/生成/后处理）的耗时、LLM token 与调用次数。

### 4.2 Dependency Graph Accuracy

**(A) Evidence**

* 现流程 Python 调用抽取只在 `callee_name` 属于已知 `components` 时记录，且组件集合看起来是“文件内抽取结果”；跨文件解析与未解析引用被弱化或丢失（你提供的阶段 2.2/2.3）。

**(B) Inference — 推荐默认策略（质量优先）**

1. **图分层**：至少维护三种图，避免“用一个不靠谱的 call 图承载全部任务”。

   * `Import/Module Graph`（高可靠）：文件/包 import 关系
   * `Symbol Reference Graph`（中可靠）：标识符引用（定义/引用）
   * `Call Graph`（分置信度）：可解析 calls + unresolved calls
2. **消歧策略**：

   * 先做 name binding（import alias、同名遮蔽、作用域）
   * 解析失败不丢弃：记录 unresolved + evidence_ref
3. **语言分级解析器（Quality-first）**：

   * TS/JS：优先 TypeScript compiler/tsserver 生成符号与引用（比 tree-sitter 仅语法更可靠）。
   * Python：可选 pyright/mypy 做类型与引用解析；若不可用则做 import 表 + 属性访问启发式（confidence=low/medium）。
   * Go：go/packages/ssa；Java：JDT/javac 插件；Rust：rust-analyzer（这些属于可插拔 indexer）。

**(C) Alternatives（至少三角度）**

* 角度 1（高可靠低成本）：只用 import 图 + 导出 API（足以做模块树骨架）。
* 角度 2（折中）：call 图只解析“同文件 + 明确导入的符号”，其他归为 unresolved。
* 角度 3（高精度）：用语义索引（SCIP/LSIF/Kythe 类）作为统一事实源。([GitHub][2])

**失败模式与回退**

* **构建配置缺失**（TS/Java/C++ 无法完整解析）：回退到 tree-sitter + import/路径启发式；confidence 降级。
* **动态特性**（Python 反射、JS 动态 import）：明确标注无法静态解析的边，文档中禁止断言行为，只能描述“可能路径”。
* **图过密**（影响聚类）：对边做类型加权与采样（例如优先 exports/imports/接口实现边）。

**Metrics & evaluation plan**

* 边质量：抽样核对（边的 evidence_ref 定位是否确实存在引用/调用）。
* unresolved 趋势：按语言/目录分桶统计，用于定位解析薄弱点。
* 聚类影响：比较“仅 import 图”和“引入 calls/ref 后”的模块稳定性差异（见 4.3）。

### 4.3 Clustering & Naming

**(A) Evidence**

* 现流程聚类由 LLM 基于“核心组件”做 grouping，模块树解析与命名缺少稳定性约束；已知失败模式包括树乱、命名差、漏组件（你给的 failure modes + 阶段 3）。

**(B) Inference — 推荐默认策略（质量优先）**

1. **三段式聚类**（稳定骨架 → 局部优化 → 语义命名）：

   * Step 1：目录/包先验切分（按语言包规则识别 package root）
   * Step 2：SCC 收缩（保证强耦合循环组件同簇）
   * Step 3：社区发现（Leiden/Louvain 之一）在“模块级加权图”上做凝聚，输出初始簇
   * Step 4：LLM 只做：

     * 生成模块名/简介/职责边界
     * 对少数“跨簇高耦合边”提出合并/拆分建议（必须给证据：跨簇边列表）
2. **命名规范（强约束）**

   * 模块 `path` 使用稳定可预测规则（例如基于目录 + 职责短名），禁止随意“文学化命名”。
   * 输出同时包含：`module_id`（稳定）、`title`（展示名）、`path`（文档路径）、`aliases`（旧名/同义名）。
3. **确定性（Determinism）策略**

   * 图算法固定随机种子 + 稳定排序
   * LLM 输出必须 schema 校验通过；同输入重复运行结果一致（通过 canonicalization：排序、去重、ID 规则固定）

**(C) Alternatives（至少三角度）**

* 角度 1（最稳）：纯目录/包树 + 少量手工规则（不会乱，但语义差）。
* 角度 2（折中）：目录树为主，社区发现只用于“合并/拆分相邻目录”。
* 角度 3（更语义）：先做“功能主题 embedding 聚类”，再用依赖图做约束校正（适合注释丰富的仓库）。

**失败模式与回退**

* **模块树过深/过碎**：回退到“按 API surface 合并”：把同一导出入口/同一包的 leaf 合并。
* **命名不稳定**：引入“命名冻结”：若模块成员集合 hash 未变，则复用旧 title/path。
* **LLM 误改边界**：只允许在“证据充分”的边界上改动（跨簇高权重边 + 双向耦合）。

**Metrics & evaluation plan**

* 结构质量：模块内凝聚/模块间耦合（用边权重求比值与趋势）。
* 稳定性：对同一 commit 多次运行，计算树相似度（例如：叶节点聚类的一致性、路径稳定）。
* 可理解性：维护者抽样打分（定性问卷：命名是否贴合、边界是否合理）。

### 4.4 Recursive Generation & Consistency

**(A) Evidence**

* 现流程叶子模块生成直接塞源码，token 超限时让 LLM 决定分裂；缺少全局术语表、重复控制与“证据引用”硬规则（阶段 4 描述 + failure modes）。

**(B) Inference — 推荐默认策略（质量优先）**

1. **证据驱动写作协议**（强制格式，见第 7 节 prompt）：

   * 文档必须分为：**(A) Evidence / (B) Inference / (C) Alternatives / (D) Limits**
   * 任何“代码行为断言”必须引用 `symbol_id` 或 `evidence_ref`
   * 任何“不确定”必须显式标注并给验证步骤（例如：grep 关键字/运行单测/定位入口文件）
2. **递归拆分策略从“LLM 自由分裂”改为“图边界拆分 + LLM 命名”**

   * 优先按依赖边界切：把跨边最少的切割作为子模块候选（graph cut / boundary edges）
   * LLM 只对候选切分做选择（受 schema 约束），并解释每个切分依据的边（证据）。
3. **一致性资产**

   * 全局术语表、全局链接 map、全局命名规则作为“共享只读上下文”注入所有生成调用。
4. **缓存**

   * 叶模块缓存粒度到 module + 组件成员集合 hash
   * 上游（符号表/图/聚类）改变才触发下游重建

**(C) Alternatives**

* 快速模式：只生成模块概览，不生成每个符号细节（适合先搭框架）。
* 省钱模式：概览用大模型，细节用小模型 + 模板化渲染（从符号卡片填充）。
* 高一致性模式：生成后加一轮“Consistency Pass”（只允许改链接/术语/重复，不允许改技术事实）。

**失败模式与回退**

* **证据不足导致文档太空**：回退到“提取更多证据片段”（扩大 retrieval 半径：相邻依赖模块）。
* **重复仍然多**：启用跨文档去重（embedding 近重复检测 → 自动改为引用链接）。
* **幻觉**：在后处理加入“事实核对器”（见 9 节：抽样核对断言与符号表是否一致）。

**Metrics & evaluation plan**

* 断言引用覆盖率：统计文档中行为断言是否附证据引用。
* 重复率：文档段落 embedding 相似度聚类，统计重复簇规模趋势。
* 正确性抽检：抽样断言 → 自动定位 symbol_id → 对比签名/位置/调用边是否存在。

### 4.5 Post-processing & Site Build

**(A) Evidence**

* 现流程后处理包含 FIX_MATH/FIX_MERMAID/FIX_LINKS，但缺少“lint 编译”与“构建门禁/回归”（阶段 5 描述 + failure modes）。

**(B) Inference — 推荐默认策略（质量优先）**

1. **Markdown AST 级处理**

   * 解析 Markdown → 统一规范化 heading/anchor → 生成全局锚点表
   * 内链校验：文件存在、heading 锚存在
2. **Mermaid 强校验**

   * 使用 mermaid-cli 编译每个 Mermaid 代码块（输出 svg 供站点使用）；失败则进入修复 prompt；再失败则降级。([GitHub][5])
3. **Math 强校验**

   * 用 KaTeX 渲染并捕获 ParseError；配置 `throwOnError=false` 的降级策略；失败进入修复 prompt。([KaTeX][6])
4. **Link 检查**

   * 本地链接：remark-validate-links（检查文件与 heading）。([GitHub][7])
   * 外链：lychee（Markdown/HTML 断链扫描，可集成 CI）。([Docs][8])
5. **回归测试**

   * 记录：坏图/坏公式/断链列表
   * 对比上一次构建：新增失败必须阻断（或至少红灯）

**(C) Alternatives**

* 只做本地链接与 Mermaid lint（最常见痛点先解决）。
* Mermaid 不预编译，只做语法 lint（更快但易漏渲染差异）。
* 外链只做白名单检查（减少噪声）。

**失败模式与回退**

* CLI 环境不稳定：提供 Docker 化的 lint 环境；或在 CI 里固定版本。
* Mermaid 编译开销：只对新增/变更页面运行（基于页面 hash）。

**Metrics & evaluation plan**

* Lint 失败数趋势：Mermaid/Math/Link 分桶统计。
* 构建稳定性：是否能做到“失败可定位、可复现、可回滚”。

**(B) Inference — 输入特征（图结构 + 目录先验 + 文本语义）**

### 5.4 伪代码：改进版 CLUSTER_MODULES + PRE_CLUSTER + VALIDATE_TREE

```python
def CLUSTER_MODULES(index, graph, config):
    """
    Input:
      index: symbol table + file map + embeddings (optional)
      graph: multi-layer graph with confidence & evidence refs
    Output:
      module_tree.json (schema v2)
    """

    # 1) Build weighted module graph (file-level first for scale)
    file_graph = build_file_graph(graph, weight_policy=config.weight_policy)

    # 2) Pre-cluster (deterministic)
    # 2.1 package/dir prior partitions
    dir_partitions = partition_by_package_roots(index.files, index.package_roots)

    # 2.2 SCC contraction to keep cycles together
    scc_groups = tarjan_scc(contract_to_subgraph(file_graph, dir_partitions))

    # 2.3 Community detection (optional but recommended for large repos)
    communities = leiden_or_louvain(
        graph=contract_graph(file_graph, scc_groups),
        seed=config.fixed_seed,
        deterministic_order=True
    )

    # 3) Lift file clusters to symbol clusters (attach symbols to their defining files)
    clusters = attach_symbols(index.symbols, communities)

    # 4) LLM naming + boundary adjustment (constrained)
    #    Provide only: cluster summaries, top boundary edges, representative symbols (cards)
    llm_input = build_cluster_naming_input(index, clusters, graph, budget=config.llm_budget)
    llm_output = LLM_CLUSTER_NAMING(llm_input, schema="codewiki.module_tree.v2")

    # 5) Validate and canonicalize
    tree = validate_schema(llm_output)
    tree = canonicalize_tree(tree)  # sort modules/members, normalize paths, stable IDs
    validate_tree_invariants(tree, index)

    return tree


def VALIDATE_TREE(tree, index):
    # Invariants:
    # - every public_api_symbol must be assigned to exactly one module
    # - every symbol appears at most once globally
    # - paths unique, module_ids stable, no cycles in tree
    # - extra top-level modules exist (Getting Started/Tutorial/Best Practices)
    errors = []
    errors += check_uniqueness(tree.members.symbols)
    errors += check_public_api_coverage(tree, index.public_api_symbols)
    errors += check_path_uniqueness(tree)
    errors += check_required_modules(tree)
    if errors:
        raise TreeValidationError(errors)
```

## 6. 文档生成改进（重点）

### 6.1 “证据驱动写作”准则（必须落地为硬规则）

**(B) Inference**

* 文档中每条“行为断言”（例如：某函数会写数据库/会发网络请求/会调用某组件）必须附 `证据引用`：

  * `symbol_id`（定义处）或 `evidence_ref`（引用/调用处）
* 若证据不足：

  * 只能写“我们观察到/索引显示…”，并在 **(D) Limits** 给出验证步骤
* 示例代码：

  * 优先来自仓库现有示例/测试/README；若必须生成“合成示例”，必须标注为推断并说明未在仓库中发现同等示例。

---

### 6.2 递归拆分策略（token 预算、深度上限、依赖边界切分）

**(B) Inference**

* 不再用“token 超限 → LLM 自由拆分”。改为：

  1. 先按图边界给出 2–3 个候选切分（deterministic）
  2. LLM 只选择其一并做命名，且必须列出每个子模块的 boundary_edges 作为依据
* 深度上限/预算不在此硬编码；由配置控制，并在日志里记录每次拆分原因（便于回归）。

---

### 6.3 缓存与增量构建策略

**(B) Inference**

* 缓存键建议包含：

  * `module_members_hash`（成员符号列表 + 关键边摘要 hash）
  * `index_version`（符号表/图版本）
  * `prompt_version`
  * `model_version`
* 增量策略：

  * 只要 `module_members_hash` 不变，就复用该模块文档与其子页面
  * 聚类树若仅 title/path 变化，细节页可不重建（但链接 map 需要重算）

### 6.4 伪代码：GENERATE_MODULE / GENERATE_LEAF_MODULE / RETRIEVE_CONTEXT

```python
def RETRIEVE_CONTEXT(module_spec, index, graph, retrieval_cfg):
    """
    Build a minimal, evidence-rich context pack for LLM.
    """
    # 1) Always include module metadata + member symbol cards
    symbol_cards = [index.symbol_cards[sid] for sid in module_spec.members.symbols]

    # 2) Include boundary edges summary (top by weight/confidence)
    boundary = summarize_boundary_edges(graph, module_spec, policy=retrieval_cfg.boundary_policy)

    # 3) Include representative callsites evidence (small snippets w/ line ranges)
    callsites = pick_representative_callsites(graph, module_spec, k=retrieval_cfg.callsites_k)

    # 4) Pull related README/config paragraphs if relevant
    docs = retrieve_docs(index, query=module_spec.title, k=retrieval_cfg.docs_k)

    return {
        "module": module_spec,
        "symbol_cards": symbol_cards,
        "boundary_edges": boundary,
        "callsites": callsites,
        "repo_docs": docs,
        "global_assets": load_global_assets()  # glossary, link_map, style invariants
    }


def GENERATE_MODULE(module_spec, index, graph, cfg):
    if cache_hit(module_spec, cfg):
        return load_cache(module_spec, cfg)

    ctx = RETRIEVE_CONTEXT(module_spec, index, graph, cfg.retrieval)

    if exceeds_budget(ctx, cfg.llm_budget) and cfg.allow_split:
        # deterministic split candidates from graph cut / directory boundaries
        candidates = propose_splits(module_spec, graph, cfg.split_policy)
        decision = LLM_SPLIT_DECISION(ctx, candidates, schema="codewiki.split.v1")
        submodules = materialize_split(module_spec, decision)
        docs = []
        for sm in submodules:
            docs.append(GENERATE_MODULE(sm, index, graph, cfg))
        overview = GENERATE_OVERVIEW(module_spec, ctx, submodules, cfg)
        return merge_docs(overview, docs)

    # leaf
    leaf_doc = GENERATE_LEAF_MODULE(ctx, cfg)
    return post_process_and_cache(leaf_doc, cfg)


def GENERATE_LEAF_MODULE(ctx, cfg):
    # Must follow: Evidence / Inference / Alternatives / Limits
    md = LLM_LEAF_DOC(ctx, prompt=cfg.prompts.leaf, schema="codewiki.leaf_doc.v1")
    return md
```

## 7. Prompt/Agent 模板库（可复制使用）

> 这些模板的共同目标：**确定性输出 + 可验证证据引用 + 可自动 lint**。
> 注意：下方模板刻意避免“让模型自由发挥”，而是让它在 schema 与证据约束下写作。

### 7.1 聚类 Prompt（JSON schema + 约束 + 示例）

**System/Developer 侧硬约束（建议）**

* 只允许输出 JSON（无额外文本）
* 必须符合 schema
* 必须按字典序排序 `children` 与 `members.symbols/files`
* 若证据不足必须写入 `limits[]`

**Input（给 LLM 的 payload 建议结构）**

### 7.2 叶子模块文档 Prompt（含：证据/推断/替代/限制格式）

**硬规则（写入 prompt）**

* 必须输出 Markdown
* 必须包含四个一级标题：`A. Evidence`、`B. Inference`、`C. Alternatives`、`D. Limits`
* `B. Inference` 中每条行为断言必须引用 `symbol_id` 或 `evidence_ref`（用统一格式）
* 不得凭空编造用法示例；若示例是推断，必须在示例标题标注“推断”并解释缺少什么证据

**Prompt 模板**

```text
你是“证据驱动”的代码文档工程师。为一个叶子模块生成 Markdown 文档。

输入包含：
- module_spec（模块成员与边界）
- symbol_cards（每个符号的签名/注释/证据位置）
- boundary_edges / callsites（带 evidence_refs）
- repo_docs（README/配置段落，带来源）

输出要求（强制）：
1) 严格按以下结构输出（不要添加其它一级标题）：
   # <模块标题>
   ## A. Evidence
   ## B. Inference
   ## C. Alternatives
   ## D. Limits

2) A. Evidence：只陈述可直接从输入证据确认的事实，并在每条后附证据引用：
   证据引用格式：
   - [symbol:<symbol_id>]
   - [ref:<file>:<start_line>-<end_line>]

3) B. Inference：可以做合理推断，但必须：
   - 每条推断说明“基于哪些证据”
   - 不能断言运行时行为（网络/IO/副作用）除非证据明确
   - 对不确定之处用“可能/推测”，并把验证步骤放到 D. Limits

4) C. Alternatives：给出至少 2 个实现/设计替代解释（例如：为什么这样组织模块、可能的边界选择）。
5) D. Limits：列出本页可能不完整/不确定之处，并给出可执行的验证步骤（具体到文件/符号/搜索关键字）。

注意：不要复述全局术语表里的定义；应链接到术语表锚点（由 link_map 提供）。
```

## 9. 评估与回归测试计划

### 9.1 覆盖率

**采集方法（不造数）**

* 组件覆盖：`documented_symbol_ids / total_symbol_ids`（按语言/目录分桶）
* 关键路径覆盖：入口点集合 → 依赖路径是否在 tutorial/模块概览中出现（以 symbol_id 引用计）

### 9.2 正确性（与代码一致性抽检）

* 静态一致性：

  * 文档引用的 symbol_id 是否存在于符号表
  * 文档宣称的签名/文件位置是否一致
* 行为断言抽检：

  * 随机抽取断言句 → 查看其 evidence_ref 是否指向真实调用/引用位置
  * 对“外部副作用”断言（网络/IO）提高抽检优先级（高风险）

### 9.3 一致性（链接/命名/重复率）

* Link lint：本地链接/锚点 0 断链；外链生成报告（不一定阻断，可白名单）。([GitHub][7])
* 命名 lint：模块 path 是否符合规范、是否重复、是否漂移（成员 hash 未变但 title 大幅变化则提示）。
* 重复率：embedding 近重复段落聚类 → 报告“应合并/应改为引用”的列表。

### 9.4 性能（阶段耗时与 token 用量分解）

* 记录每阶段：

  * I/O 扫描与解析耗时
  * 索引构建耗时
  * 聚类耗时
  * 文档生成调用次数与 token（按模块分桶）
  * 后处理 lint 耗时
* 对比策略：同 commit 下分别跑“质量优先/快省模式”，用日志而不是臆测来决定取舍。

## 10. 风险、权衡与落地路线图

### 10.1 风险

* 多语言解析误差：构建配置缺失/动态特性导致图不完整。
* LLM 不稳定：即使 schema 约束也可能输出漂移，需要重试与 canonicalization。
* 索引漂移：增量更新若 hash 粒度不当会造成“该重建不重建/不该重建却重建”。
* 外部引用失效：外链断链不可避免，需要白名单与缓存策略。

### 10.2 权衡（质量优先下的最小成本优化路径）

* 先把“坏图”变成“可解释的中等好图”，再追求更高精度：

  1. 先做符号表 + import 图 + 导出 API（高收益低风险）
  2. 再引入 calls/ref（分置信度）
  3. 最后才考虑 SCIP/LSIF 级别的语义统一（投入大但收益长期）([GitHub][2])
* LLM 使用策略：

  * 把 LLM 从“做决定”改成“写说明/命名/在候选中选择”会显著提升可控性与可回归性。

## Appendix B: 术语表（首次中英并列）

* 符号表（Symbol Table）：记录符号定义位置、签名、文档与属性的索引。
* 组件卡片（Symbol Card / Component Card）：面向生成与检索的轻量符号摘要，包含证据引用。
* 证据引用（Evidence Reference）：指向源码位置的可追溯指针（文件 + 行列范围）。
* 未解析引用（Unresolved Reference）：静态分析无法解析到具体符号的引用/调用记录。
* 预聚类（Pre-cluster）：在调用 LLM 前用图算法/目录规则先定结构骨架。
* 确定性（Determinism）：同一输入在同一版本下输出完全一致，便于回归与增量。
* Lint 门禁（Lint Gate）：构建阶段强校验，失败必须修复或降级，不允许静默通过。
* 增量构建（Incremental Build）：仅重建受变更影响的模块/页面。

---

**审计要点（权衡/风险简述）**

* 质量优先的核心是：先把“事实层”（索引/图）做对、做可追溯，再让 LLM 写作。
* 最大风险是多语言语义解析成本；建议按语言/目录做分级落地，并用 unresolved+置信度避免“错误事实冒充正确事实”。

## 方案的核心思路

这个方案的根本哲学可以用一句话概括：**把 LLM 从"决策者"降级为"写手"，让确定性算法和可验证的数据产品承担结构性工作。** 这是整个方案最关键的思想转变。

你原来的 CodeWiki 流程本质上是"LLM 驱动"的——聚类靠 LLM 做，拆分靠 LLM 做，生成也靠 LLM 做。方案把这个范式翻转为"数据驱动 + LLM 辅助"：先用静态分析和图算法建立一个可靠的事实基础（符号表、依赖图、预聚类骨架），然后只让 LLM 在严格的 schema 约束下做命名、撰写说明、在程序给出的候选方案中做选择。这种转变解决的是你原系统最根本的问题——**不可控性和不可复现性**。

## 五层架构的逐层分析

### 离线索引层（Index Layer）——最大的增量价值

这一层是整个方案中 ROI 最高的部分。你原来的系统直接从 AST 抽取组件然后塞进 prompt，中间没有"索引产物"这个概念。方案引入的"符号表 + 组件卡片 + 向量库"三级索引结构，本质上是在源码和 LLM 之间插入了一个**可审计、可缓存、可检索的中间层**。

这解决了一个很现实的问题：当仓库有上百万行代码时，你不可能把所有源码都塞进 LLM 的上下文窗口。原系统的做法是靠 LLM 自己决定怎么裁剪（递归分裂），这既不可控又浪费 token。有了索引层之后，你可以做精准的"证据检索"——只把生成某个模块文档真正需要的符号卡片、关键调用片段、相关 README 段落喂给 LLM，这就是方案里说的 **Context Pack** 思路。

不过这里有一个需要注意的**实现复杂度陷阱**：方案建议的分层嵌入策略（README 段落、文件摘要、符号卡片、调用片段四层）看起来很完美，但在实际工程中维护四层 embedding 的一致性和增量更新是非常繁琐的。我的建议是先只做符号卡片这一层的 embedding（因为它是最结构化、最容易保持一致性的），等跑通了再逐步加层。

### 图构建与消歧层（Graph Layer）——方向正确但代价需要评估

方案提出的统一组件 schema 设计得很好，特别是两个关键改进：一是**保留未解析引用**（unresolved references），二是给每条边加 **confidence_level 和 evidence_refs**。

你原来的系统有一个很要命的问题：`callee_name in components` 这个条件意味着所有解析不到的调用都被直接丢弃了。在 Python 这种动态语言里，大量调用是跨文件的、通过 import alias 的、甚至是动态分派的，这些调用全部丢掉之后，你的依赖图就变成了一个严重低估真实耦合的"稀疏图"，在这样的图上做聚类自然效果差。

方案的"保留 unresolved + 标置信度"策略是对的，但这里有一个**微妙的权衡**：unresolved 边太多（比如 Python 里可能占 30-50%）会给后续聚类引入大量噪声。方案提到了这个风险但没有给出具体的处理策略。一个实用的做法是：在聚类阶段对 unresolved 边做**降权而非等权**处理，比如 high confidence 边权重 1.0，medium 0.5，low/unresolved 0.1，这样既不丢信息又不被噪声主导。

另外，方案建议的"语言分级解析器"策略（TS 用 tsc、Python 用 pyright、Go 用 go/packages）虽然理想，但这意味着你需要为每种语言维护一套独立的 indexer，工程量非常大。方案也给了分级替代方案（角度 1/2/3），我觉得**角度 2 是最务实的起点**：对强类型语言用官方分析器，Python 只做 import 表 + 属性访问启发式。

### 聚类层（Clustering Layer）——方案最精彩的部分

这是我认为整个方案设计最巧妙的地方。原系统的聚类完全交给 LLM，结果就是"同一个仓库跑两次得到两棵完全不同的模块树"。方案的"三段式聚类"（目录先验 → SCC 收缩 → 社区发现 → LLM 命名）把这个问题拆解得非常漂亮。

为什么这个设计好？因为它利用了不同算法各自的优势：目录/包结构给出了**人类意图的先验**（开发者已经按目录组织了代码），SCC 保证了**循环依赖必须在同一模块**（这是硬约束），社区发现（Leiden）给出了**基于图拓扑的最优分组**，最后 LLM 只做人类不擅长自动化的事——**给簇起一个好名字、写一句好描述**。

这里我想提醒你注意 Leiden 算法的一个特性：它的 resolution 参数会显著影响聚类粒度。resolution 太大会导致模块过碎（每个文件一个模块），太小会导致一切都混在一起。方案提到了"参数需调"但没给具体策略。一个实用的做法是：**用多个 resolution 值跑一遍，然后选 modularity 增益拐点对应的那个值**，这样既有理论依据又可复现。

### 生成层（Generation Layer）——"证据驱动写作"是好概念但落地要谨慎

方案提出的"两阶段写作"（先概览再细节）和"证据驱动"格式（A/B/C/D 四段式）是针对原系统幻觉和重复问题的直接对策。

两阶段写作的好处很明确：先有概览才能建立全局术语表和链接 map，后面的叶子模块文档就可以引用而不是重复。这解决了原系统"每个叶子模块都在重新解释公共概念"的问题。

但 **A/B/C/D 四段式格式我认为需要斟酌**。对于面向开发者的代码文档来说，这种格式太学术化了。开发者查文档通常只想快速了解"这个模块干什么、怎么用、有什么坑"，把内容分成 Evidence/Inference/Alternatives/Limits 四个块会严重影响可读性。这种格式更适合作为**内部质量保证的中间产物**，最终面向用户的文档应该转换成更传统的技术文档结构（Overview → API Reference → Examples → Notes）。你可以在生成过程中要求 LLM 按 A/B/C/D 思考，但输出时做一次格式转换。

### 后处理层（Post Layer）——从"修修补补"到"质量门禁"

方案把后处理从"best effort 修复"升级为"lint + 修复 + 降级 + 回归"的四级策略，这是非常工程化的思路。特别是"降级而非静默通过"的原则——Mermaid 编译失败就降级为代码块而不是默默生成坏图，这比原系统好得多。

mermaid-cli 预编译为 SVG 的策略也很实用，因为 Mermaid 在浏览器端渲染经常有兼容性问题，预编译为 SVG 可以一劳永逸。

## 方案的整体优缺点

**最大的优点**是它建立了一个**可观测、可回归、可增量**的工程体系。原系统像一个黑盒（源码进去、文档出来、中间不知道发生了什么），新方案让每一步都有可检查的中间产物（符号表、带置信度的图、schema 校验过的模块树、lint 日志）。

**最大的风险**是**实现复杂度**。这个方案涉及的技术栈非常广：多语言 AST/语义分析、图算法（Tarjan SCC、Leiden）、向量检索、JSON Schema 校验、mermaid-cli/KaTeX/lychee 等工具链集成、增量缓存系统。如果试图一次性实现所有这些，很容易陷入"永远在搭基础设施、永远没有可用产出"的困境。

方案给出的里程碑路线图（5 个里程碑）是合理的，但我建议你用一个更激进的**最小可行改进**策略：先只做里程碑 1 中的符号表 + import 图，加上里程碑 3 中的"目录先验 + LLM 命名"，这两个改动单独就能解决原系统"模块树乱"和"漏组件"两个最痛的问题，而且实现量可控。等这个跑通了再逐步加 calls/ref 解析、社区发现、证据驱动写作等更复杂的特性。

## 一个值得思考的问题

方案在多处强调"确定性"（同输入同输出），这对可回归性很重要，但也有一个隐含代价：**你放弃了 LLM 在不同运行中可能给出更好答案的可能性**。比如模块命名，第一次跑出来的名字可能不如第三次的好，但为了确定性你会锁定第一次的结果。方案通过"命名冻结"策略来处理这个问题（成员 hash 不变就复用旧名），但这意味着一个不好的名字可能会被长期锁定。你可能需要一个"手动解冻"或"定期重评估"的机制来平衡确定性和质量迭代。

总的来说，这是一个非常扎实的升级方案，核心思路完全正确。关键是控制好落地节奏，不要试图一步到位。
