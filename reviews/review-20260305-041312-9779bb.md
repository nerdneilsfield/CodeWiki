## Critical

- **Missing committed agent runbook + CI gates**
  - Location: repo root (no committed `AGENTS.md` / `CLAUDE.md`, no `.github/workflows/`)
  - Severity: critical
  - Category: Holistic
  - Description: There is no canonical “how to work here” guide for humans/agents, and no CI enforcement for lint/type/tests. This makes regressions likely and slows down automated contributions.
  - Suggested fix: Commit a minimal root `AGENTS.md` (conventions, boundaries, common commands, test/lint/type workflow) and add CI jobs for `ruff`, `mypy`, `pytest`, plus a small smoke check (e.g., import + `--help` for the CLI/web entrypoint).

## High

- **Doc generation can fail hard due to unconditional index build**
  - Location: `codewiki/src/be/documentation_generator.py:694` (approx)
  - Severity: high
  - Category: Diff
  - Description: Index building is now an unconditional step in the doc generation flow; if `IndexBuilder` import/build raises (optional deps missing, parser issues, unexpected file errors), the entire documentation generation run can fail.
  - Suggested fix: Wrap the `IndexBuilder` import/build in `try/except` and continue with `self.index_products = None` (or an empty `IndexProducts`), logging the exception with `exc_info=True`.

- **Dependency “source of truth” is split, Docker installs differently than package**
  - Location: `pyproject.toml`, `requirements.txt`, `docker/Dockerfile`
  - Severity: high
  - Category: Holistic
  - Description: `pyproject.toml` and `requirements.txt` appear to represent different dependency sets (Docker installs `requirements.txt` directly and does not install the package). This can cause drift between local/dev, Docker, and CI, and can mask packaging/import issues.
  - Suggested fix: Choose one canonical dependency workflow (e.g., `pyproject.toml` + lockfile, or `requirements.in` → compiled lock), and make Docker/CI install from the same source (ideally `pip install .` plus lock/constraints).

- **Credential/config footguns: dummy API key patterns + weak validation**
  - Location: `codewiki/src/config.py`, `docker/env.example`
  - Severity: high
  - Category: Holistic, Diff
  - Description: A placeholder like `LLM_API_KEY=sk-1234` both looks like a real secret format (trips scanners/confuses users) and encourages “runs with placeholder creds”. Web-app mode relies on env vars but does not strongly validate that a real key is configured.
  - Suggested fix: Replace placeholders with `REPLACE_ME`/`YOUR_API_KEY`, fail fast when key is missing/placeholder, and centralize env parsing/validation (e.g., `pydantic-settings`) for web paths.

## Medium

- **`include_patterns` / `exclude_patterns` accepted but not applied**
  - Location: `codewiki/src/be/index/index_builder.py:54` and `codewiki/src/be/index/index_builder.py:126` (approx)
  - Severity: medium
  - Category: Diff
  - Description: Patterns are plumbed through from the doc generator but never used during file discovery, so configuration is silently ignored.
  - Suggested fix: Apply the patterns when filtering discovered files (likely in `_discover_files()`), and add tests proving inclusion/exclusion behavior.

- **Index file discovery ordering may be non-deterministic**
  - Location: `codewiki/src/be/index/index_builder.py:128` (approx)
  - Severity: medium
  - Category: Diff
  - Description: `_discover_files()` returns files in filesystem traversal order, which can vary across OS/filesystem. This risks non-deterministic symbol/import ordering and flaky outputs.
  - Suggested fix: Sort discovered results (e.g., by `rel_path`) before returning.

- **Import edges anchored to an arbitrary “first symbol”**
  - Location: `codewiki/src/be/index/index_builder.py:152` (approx)
  - Severity: medium
  - Category: Diff
  - Description: Import edges are anchored to `from_syms[0].symbol_id` (first symbol found in a file), which can misattribute “file imports X” edges to an arbitrary symbol and distort component cards/graphs.
  - Suggested fix: Introduce a stable module/file-level node (or store `from_file` separately on edges) and anchor import edges to that, not to an incidental symbol.

- **Calls into private analyzer API**
  - Location: `codewiki/src/be/index/index_builder.py:120` (approx)
  - Severity: medium
  - Category: Diff
  - Description: `CallGraphAnalyzer._analyze_code_file` is a private method; calling it directly is brittle and makes upgrades/refactors riskier.
  - Suggested fix: Add or use a public API wrapper and call that instead of a `_...` method.

- **Potential `symbol_id` collisions in generic adapter**
  - Location: `codewiki/src/be/index/adapters/generic_adapter.py:41` (approx)
  - Severity: medium
  - Category: Diff
  - Description: `symbol_id` is built from `{lang}:{rel_path}#{node.name}({kind})`, which can collide for multiple same-named symbols in a file/scope. Downstream “last write wins” in symbol tables can drop/overwrite data.
  - Suggested fix: Add uniqueness (e.g., `node.id`, `start_line`, or a stable hash) to `symbol_id`.

- **Docstring summary can blow up in size; potential prompt-injection surface**
  - Location: `codewiki/src/be/index/component_card.py:30` (approx)
  - Severity: medium
  - Category: Diff
  - Description: Docstring truncation is sentence-count only; a single long sentence (or no punctuation) can produce a very large `docstring_summary`. This increases prompt size and makes it easier for untrusted docstrings to reach LLM consumers unbounded.
  - Suggested fix: Add a hard cap on summary length (chars/tokens), normalize whitespace, and consider basic sanitization (e.g., cap repeated newlines).

- **Risky broad monkey-patch of `pydantic_ai` internals**
  - Location: `codewiki/src/be/agent_orchestrator.py`
  - Severity: medium
  - Category: Holistic
  - Description: A broad monkey-patch of a private function (with very wide exception handling) increases upgrade risk and can hide failures.
  - Suggested fix: Gate by version, log explicitly when patching, add a focused unit test for the upstream bug scenario, and prefer an upstream fix or public extension hook.

- **Inconsistent observability/error reporting**
  - Location: `codewiki/src/fe/*.py`, `codewiki/src/be/dependency_analyzer/analysis/cloning.py`, `codewiki/src/be/utils.py`
  - Severity: medium
  - Category: Holistic
  - Description: Logging/telemetry appears inconsistent (multiple `print(...)` paths; partially commented-out instrumentation). This makes production debugging and agent-run traceability harder.
  - Suggested fix: Standardize on `logging` (structured if possible), remove/replace `print`, add correlation IDs for web flows, and either fully enable or remove the partially wired instrumentation to avoid “half-on” behavior.

- **Perf tests likely to be flaky in CI**
  - Location: `tests/test_perf_utils.py`, `tests/test_perf_docs_fixer.py`, `tests/test_perf_parallel_analysis.py`
  - Severity: medium
  - Category: Holistic
  - Description: Wall-clock assertions (tight time budgets / parallelism thresholds) vary across machines and CI load; they appear to run in the default suite.
  - Suggested fix: Mark as `@pytest.mark.perf` and exclude from default CI (or make deterministic via mocking timers / asserting relative behavior without strict time bounds).

- **Docker documentation drift**
  - Location: `docker/DOCKER_README.md`, `docker/docker-compose.yml`
  - Severity: medium
  - Category: Holistic
  - Description: Docker docs appear out of sync with actual compose/Docker behavior (e.g., differing `PYTHONPATH` notes), increasing onboarding friction.
  - Suggested fix: Align docs with current config and add a lightweight CI check to prevent drift.

- **Project metadata / URLs inconsistent**
  - Location: `README.md`, `pyproject.toml`
  - Severity: medium
  - Category: Holistic
  - Description: Install-from-source instructions and `project.urls` placeholders reduce trust and reproducibility.
  - Suggested fix: Update README + `project.urls` to the canonical repo/docs/issue tracker; add a short release/versioning note.

- **Web entrypoint relies on `sys.path` injection / `PYTHONPATH` over packaging**
  - Location: `codewiki/run_web_app.py`, `docker/Dockerfile`
  - Severity: medium
  - Category: Holistic
  - Description: Running via `PYTHONPATH` and manual path manipulation can mask import/package issues and complicate tooling (mypy/IDEs) and reuse as a library.
  - Suggested fix: Install the package in Docker (`pip install .` or `pip install -e .` in dev images) and run via module/console script instead of path edits.
